# 協働スタイル

k-pdf3 で確立されたユーザーとの協働方針を k-file にも適用する。

## ユーザー像
- 法律実務家 (sk21.lawyer@gmail.com)、プログラミングは素人前提
- ただし「素人」=「適当でいい」ではない。実装の品質には妥協しない人
- 業務 (法律事務) で実際に使うものを作っている。机上ではなく実機検証を重視

## 基本方針

### 1. 素人前提・先回り提案
- 専門用語を当然視せず、初出時に短く説明する
- 「ここで A と B の選択がある、A が業務向き・B が柔軟性向き」のように、選択肢と推奨を先に出す
- ユーザーが質問する前に、当然次に出てくる懸念点を潰しておく

### 2. 趣旨読み取り
- 表面の文言ではなく、ユーザーが「本当は何を解決したいか」を考える
- 例: 「これをこう変えて」 → 「なぜそう変えたいか」を踏まえると別実装が適切なことがある
- 不明な時は実装前に「これは X のためという理解で OK?」と確認する

### 3. 長期目的を見失わない (architecture-first)
- 目の前のバグ修正で構造を壊さない
- 「今は動くがいずれ詰む」設計を見つけたら、その場で警告する (実装は別判断)
- 機能追加は積み上げではなく構造に組み込む

### 4. 「分かりやすい説明」と「妥協した実装」を混同しない
- ユーザー向け文章は平易に
- コード・設計は妥協なし
- 「素人だから雑でいい」は禁忌

## 振る舞いルール

### HANDOVER.md は明示依頼時のみ更新
- マイルストーン達成、β タグを切る、機能追加完了の都度は更新しない
- 「HANDOVER 更新して」「引き継ぎ書いて」と言われた時だけ
- 理由: 自動更新だと差分が膨大になり、ユーザーがレビューできなくなる

### 報告 → 原因 → 修正 → 検証 を 30 分サイクルで
- バグ報告を受けたら、まず原因仮説を 1-2 案立てる
- 1-2 commit で最小修正 (大きな refactor は別タスク)
- 修正後は (β リリースなら) 配布、(機能なら) 動作確認の手順を提示
- 機能追加より「業務凍結級バグの修正」を優先

### バージョンバンプは慎重に
- 詳細は `VERSION-BUMPING.md`
- マイルストーン (M1, M2, ...) と alpha/beta/stable は明確に対応させる
- β タグは「業務並走テスト中」を意味する公式 signal

### UI 方針からは絶対に外れない
- Win95/98 風レトロ業務アプリ感を最優先 (`UI-PRINCIPLES.md`)
- placeholder 段階でも雰囲気を保つ
- モダン Flat UI / マテリアル / Tailwind 風が「便利そう」に見えても採用しない

### 提案する時は選択肢を出す
- 「A 案でいきます」より「A 案 (推奨、理由 X) / B 案 (理由 Y) どちらにしますか」
- ユーザーが判断したい局面 (UX 細部、業務フィット、優先順位) では特に重要

## 過去の失敗パターン (k-pdf3 由来、繰り返さない)

- **βタグ毎の自動 HANDOVER 更新** → 差分肥大で読めなくなった (2026-05-12 にユーザーから明示停止指示)
- **解像度を下げる方向の最適化** → 印刷品質を犠牲にする提案は却下される (β.31 で 900dpi に上げて満足した直後)
- **過剰実装** → β.88 で「FAX 印刷の保険」で全ページ 900dpi raster してしまい黒紙事故。Phase 系の積み増し時は範囲を明確に
- **モダン UI への流出** → 何度か出かかったが都度引き戻された。最初から避ける
